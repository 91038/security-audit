# 코드 취약점 점검 체크리스트

스캐너가 잡는 패턴 외에, 아래 영역은 **직접 코드를 읽어** 확인한다. 바이브코딩
앱에서 실제로 사고가 나는 곳은 거의 다 여기다. 각 항목은 "공격자 관점"에서
생각한다: *내가 악의적 사용자라면 이걸 어떻게 악용할까?*

## 목차
1. 인증 (Authentication)
2. 접근제어 / 인가 (Authorization)
3. 시크릿 / 자격증명 관리
4. 인젝션 (SQL / NoSQL / OS Command)
5. XSS 및 출력 인코딩
6. 입력 검증 / 신뢰 경계
7. 민감정보 노출
8. SSRF / 파일 / 리다이렉트
9. 암호화 / 난수
10. 비즈니스 로직

---

## 1. 인증 (Authentication)
- 비밀번호를 평문/약한 해시(MD5, SHA1)로 저장하는가? → bcrypt/argon2/scrypt 사용해야 함.
- 로그인 시도 제한(rate limit)·계정 잠금이 없는가? → 무차별 대입(brute force) 가능.
- JWT를 클라이언트에서만 검증하거나 `alg: none`을 허용하는가? 서명 검증을 서버에서 하는가?
- 세션 토큰이 예측 가능하거나 로그아웃 시 무효화되지 않는가?
- 비밀번호 재설정 토큰이 추측 가능하거나 만료가 없는가?
- OAuth/소셜 로그인의 `state` 검증 누락(CSRF) 여부.

## 2. 접근제어 / 인가 (Authorization) — 가장 흔하고 치명적
- **IDOR**: `/api/orders/123` 에서 ID만 바꾸면 남의 데이터가 보이는가? 서버가
  "이 리소스가 현재 사용자 소유인가"를 매 요청 검증하는가?
- 권한 체크가 **프론트엔드에만** 있고 백엔드 API에는 없는가? (UI에서 버튼만 숨김)
- 관리자 전용 엔드포인트가 일반 사용자 토큰으로 호출 가능한가?
- Supabase/Firebase: **RLS(Row Level Security)/보안 규칙**이 꺼져 있거나
  `true`로 전면 허용돼 있는가? anon 키로 전체 테이블 read/write가 되는가?
- 객체 수준뿐 아니라 **함수/필드 수준** 권한도 확인 (예: 일반 사용자가 `role` 필드 변경).

## 3. 시크릿 / 자격증명 관리
- API 키, DB 비밀번호, service_role/admin 키가 **소스코드에 하드코딩**돼 있는가?
- 서버 전용 시크릿이 **클라이언트 번들**에 포함됐는가? (Next.js: `NEXT_PUBLIC_`
  접두사 없는 키가 클라이언트에서 import됨 / Vite: `VITE_` 규칙)
- `.env` 파일이 `.gitignore`에 없거나 git 히스토리에 커밋돼 있는가?
- 로그에 토큰/비밀번호/PII가 찍히는가?

## 4. 인젝션 (SQL / NoSQL / OS Command)
- 사용자 입력을 문자열 결합으로 쿼리에 넣는가? → 파라미터라이즈드 쿼리/ORM 사용.
  - 위험: `db.query("SELECT * FROM u WHERE id=" + req.params.id)`
- NoSQL: 사용자 입력 객체를 그대로 쿼리에 넣어 `{$gt:""}` 같은 연산자 주입 가능?
- `exec`, `child_process`, `os.system` 등에 사용자 입력이 들어가는가? (OS 커맨드 인젝션)
- ORM이라도 `raw()`/`literal()` 안에 입력을 직접 넣으면 위험.

## 5. XSS 및 출력 인코딩
- React `dangerouslySetInnerHTML`, Vue `v-html`, `innerHTML`에 사용자/외부 데이터가 들어가는가?
- 서버 템플릿에서 자동 이스케이프를 끄고(`| safe`, `{{{ }}}`) 사용자 데이터를 렌더링하는가?
- 사용자가 넣은 URL을 `href`/`src`에 그대로 쓰는가? (`javascript:` 스킴 주입)
- DOM 기반 XSS: `location.hash`, `document.referrer` 등을 sink에 직접 연결.
- 완화책으로 CSP 헤더가 설정돼 있는가?

## 6. 입력 검증 / 신뢰 경계
- 모든 외부 입력(쿼리, 바디, 헤더, 파일명, 웹훅)을 **서버에서** 검증/스키마 적용하는가?
- 파일 업로드: 확장자/콘텐츠 타입/크기 검증, 실행 가능 경로 저장 금지, 파일명 정규화.
- 대량 할당(mass assignment): `User.update(req.body)` 처럼 바디 전체를 신뢰하는가?

## 7. 민감정보 노출
- API 응답이 UI에 안 쓰는 필드(비밀번호 해시, 이메일, 내부 ID, 토큰)까지 내려주는가?
- 에러 응답에 스택 트레이스·SQL·내부 경로가 노출되는가? (프로덕션 디버그 모드)
- 디렉터리 리스팅, `.git`, `.env`, 백업 파일이 웹에서 접근 가능한가?
- CORS가 `Access-Control-Allow-Origin: *` + credentials 허용으로 열려 있는가?

## 8. SSRF / 파일 / 리다이렉트
- 서버가 사용자가 준 URL로 요청을 보내는가? (이미지 프록시, 웹훅) → 내부망/메타데이터(169.254.169.254) 접근 위험.
- 경로 조작(path traversal): `../`로 의도 밖 파일 읽기/쓰기 가능?
- 오픈 리다이렉트: `?redirect=` 값을 검증 없이 그대로 이동.

## 9. 암호화 / 난수
- 보안 토큰/세션 ID에 `Math.random()` 같은 비암호학적 난수 사용?
- 약한 알고리즘(DES, ECB 모드, 하드코딩된 IV/키)?
- TLS 인증서 검증을 끄는 코드(`rejectUnauthorized: false`, `verify=False`)?

## 10. 비즈니스 로직
- 가격/수량/권한을 클라이언트가 보낸 값으로 신뢰하는가? (예: 결제 금액을 프론트에서 결정)
- 경쟁 조건(race condition): 쿠폰/잔액 동시 사용으로 중복 차감 우회?
- 워크플로 단계 건너뛰기(결제 없이 주문 완료 등)?
