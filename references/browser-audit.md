# 라이브 브라우저 점검 가이드

실행 중인 앱을 **브라우저로 직접 열어** 코드만으로는 안 보이는 런타임 취약점을
찾는다. Claude in Chrome 확장(`mcp__Claude_in_Chrome__*` 도구)이 필요하다.
도구가 없으면 사용자에게 확장 설치를 안내하고 이 단계는 건너뛴다.

> 원칙: **사용자 본인 앱**(주로 localhost/스테이징)만 점검한다. 제3자 사이트를
> 공격하거나 실제 데이터를 파괴/유출하지 않는다. 읽기/관찰 위주로 진행한다.

## 0. 페이지 목록 만들기
- 사용자에게 시작 URL(예: `http://localhost:3000`)을 받는다.
- 라우터 설정(`app/`, `pages/`, `routes`)을 코드에서 읽어 주요 라우트 목록을 만든다.
- 로그인 전/후, 일반/관리자 등 권한별로 점검할 페이지를 나눈다.

## 1. 각 페이지에서 수행
`mcp__Claude_in_Chrome__navigate`로 이동 후:

### 네트워크 (`read_network_requests`)
- **시크릿 노출**: 요청 URL 쿼리스트링이나 헤더에 API 키/토큰/비밀번호가 보이는가?
- **평문 전송**: 민감 데이터가 `http://`(비TLS)로 오가는가? 혼합 콘텐츠?
- **과다 응답**: API 응답이 화면에 안 쓰는 민감 필드(이메일, 해시, 내부 ID, 권한)까지 내려주는가?
- **인증 누락**: 인증이 필요해 보이는 API가 토큰 없이도 200을 주는가? (헤더 제거 후 재현은 아래 6번)
- **CORS 헤더**: 응답의 `Access-Control-Allow-Origin`이 `*`인지, credentials 동반인지.

### 콘솔 (`read_console_messages`)
- 노출된 스택 트레이스, 내부 경로, SQL, 디버그 로그.
- 보안 경고(혼합 콘텐츠, CSP 위반, 쿠키 경고).
- 클라이언트에 찍힌 토큰/PII.

### 응답 헤더 & 쿠키
- 보안 헤더 존재 여부: `Content-Security-Policy`, `Strict-Transport-Security`,
  `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`.
- 세션/인증 쿠키의 `HttpOnly`, `Secure`, `SameSite` 플래그.
- 서버 정보 과다 노출 헤더(`X-Powered-By`, 상세 `Server` 버전).

### 페이지 내용 (`read_page` / `get_page_text`)
- 폼에 CSRF 토큰이 있는가(상태 변경 요청).
- 숨겨진 디버그/관리 UI, 주석에 남은 자격증명/내부 정보.

## 2. 입력 지점 관찰 (안전 범위)
- 검색창·댓글·프로필 등 사용자 입력이 화면에 반사되는 곳에 무해한 마커
  (예: `zzqa123<i>`)를 넣고, 그대로 HTML로 렌더되는지(반사형 XSS 신호) 관찰.
  실제 공격 페이로드(쿠키 탈취 등)는 만들지 않는다 — 인코딩 여부만 확인.

## 3. 접근제어(IDOR) 점검 — 본인 앱 한정
- 로그인 후 보이는 객체 URL(`/api/items/42`, `?id=42`)의 ID를 바꿔
  남의 리소스가 보이는지 관찰.
- 로그아웃 상태에서 인증 페이지/엔드포인트에 직접 접근되는지 확인.
- 가능하면 두 번째 테스트 계정으로 A의 데이터가 B에게 보이는지 교차 확인.

## 4. 기록
관찰된 각 이슈를 `findings.json`에 기록한다. `location`에는 해당 URL/엔드포인트를,
`evidence`에는 실제 요청/응답/헤더 일부를 짧게 담는다(시크릿은 마스킹).
category 예: `정보 노출`, `보안 헤더/설정`, `접근제어`, `XSS`, `인증/세션`.

## 5. 스크린샷(선택)
주요 이슈 화면은 `get_screenshot`으로 캡처해 두면 보고서 신뢰도가 올라간다.
필요 시 PDF에 첨부하도록 사용자에게 제안한다.
